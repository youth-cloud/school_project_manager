# 可优化项记录

## 当前说明
- 本文件只记录“当前先不做，但后续值得继续优化”的内容
- 现在项目已完成主业务第一版、主要业务纠偏和一轮完整回归测试，优化项不再按历史阶段堆叠，而按真实优先级重新整理
- 当前最值得投入的方向已经从“补模块”转为“性能优化 + 详情体验统一 + 部署准备”

## 一、当前最高优先级

### 1. 前端性能优化
- 当前数据量很小的情况下，部分页面首屏仍会出现 `1~2s` 的 `no data` 或空白等待
- 后续应优先排查：
- 是否存在页面 `onMounted` 串行请求过多
- 是否存在详情弹窗“先开弹窗再等多次接口返回”的模式
- 是否存在无必要的全量 `options` 重复请求
- 是否存在查询条件和详情数据分开多次请求但没有并行化
- 建议优先优化的页面：
- `project_topic`
- `topic_application`
- `project_group_application`
- `project_group`
- `project_group_member`
- `stage_submission`
- `review_record`
- `defense_record`

### 2. 详情 loading / skeleton / empty 体验统一
- 当前只有部分页面补了加载态和骨架屏，整体体验还不统一
- 后续建议统一出一套规则：
- 列表首屏使用统一 loading 状态
- 详情弹窗打开后先显示 skeleton，而不是空白框
- 空列表、空附件、空详情采用统一 empty 组件文案
- “提交中 / 删除中 / 加载中”按钮状态风格统一
- 这一项做完后，前端观感会明显更像成品系统

### 3. 部署准备文档
- 当前代码层已接近可部署状态
- 后续仍需形成清晰文档：
- 后端启动方式
- 前端打包方式
- MySQL 初始化步骤
- Redis 配置说明
- 文件上传目录配置
- token、跨域、环境变量配置
- 云服务器部署步骤
- Nginx 或反向代理方案

## 二、前端产品化优化

### 1. 列表页与详情页继续通用化
- 当前很多页面已经形成统一模板，但实现仍分散
- 后续可继续抽：
- 通用分页状态
- 通用详情 loading / skeleton
- 通用状态标签映射
- 通用时间格式化
- 通用确认删除逻辑
- 通用详情卡片布局

### 2. 页面中仍可能残留零碎教学文案或过渡性表达
- 当前已做过一轮大清理
- 后续仍建议在最终交付前再统一扫一遍：
- 页头副标题
- 暂时性说明
- 教学提示文案
- 开发期标签
- 接口说明式文字

### 3. 首页与少量页面仍可继续精修视觉表现
- 当前整体风格已经可用
- 后续可继续优化：
- 首页卡片信息密度
- 快捷入口
- 管理员常用功能聚合
- 细节动画和视觉层级
- 少量页面标题区的一致性

### 4. 选择器体验仍可继续升级
- 当前很多纯 ID 已经改成名称与选择器
- 后续可继续补：
- 远程搜索
- 大数据量分页搜索
- 选项缓存
- 联动选择器
- 更明确的空态与加载态

## 三、业务规则层优化

### 1. 建组申请方案 C 仍可继续产品化
- 当前主流程已跑通：学生申请建组、教师审核、系统自动生成正式项目组
- 当前已进一步调整为：
- 申请人默认成为组长
- 其他成员可通过输入项目组 ID 直接加入
- 后续可继续补：
- 建组申请与正式项目组跳转联动
- 更清晰的审核后反馈
- 申请历史状态追踪
- 组 ID 分享入口的进一步精修

### 2. StageSubmission 仍可继续增强业务一致性
- 当前已完成提交链路第一轮校验，并在前端详情页整合附件管理
- 后续仍可补：
- `submitterId` 是否真属于该项目组成员
- `versionNo` 改为后端自动生成
- 同任务同项目组版本唯一性
- 提交状态进一步产品化
- 教师查看同一组多版本提交的差异化体验

### 3. WeeklyReport 仍可继续流程化
- 当前周报第一版已可用
- 后续仍可补：
- 学生是否属于项目组校验
- 同一学生同一周次唯一性
- 教师点评与学生提交拆成独立动作
- 周报状态流转更清晰

### 4. DefenseSchedule / DefenseRecord / ScoreRecord 仍可继续规则化
- `defense_schedule` 后续可补：
- 答辩日期与批次时间范围约束
- 同时间地点冲突校验
- 同批次顺序唯一性
- `defense_record` 后续可补：
- 评审教师资格约束
- 重复评分策略
- 总分后端自动计算
- 分项上限与权重规则
- `score_record` 后续可补：
- 同批次同学生唯一成绩约束
- `finalScore` 与 `gradeLevel` 后端自动生成
- 更明确的成绩发布状态

## 四、认证与安全层优化

### 1. 认证生命周期仍可继续增强
- 当前已完成：登录、`/me`、改密、管理员重置密码、refresh、logout、旧 token 失效控制
- 后续可继续补：
- 双 token 模型
- 多设备登录策略
- 强制下线指定用户
- 登录失败次数限制
- 风险登录提示
- 更细的 token 生命周期配置化

### 2. 用户选项接口仍可继续按场景裁剪
- 当前已经修复过“不同身份显示不一致”的问题
- 后续仍可继续优化为：
- 不同页面只返回当前页面真正需要的候选人
- 复杂场景改成分页搜索型选择器
- 避免长期依赖通用全量 options 接口

### 3. 操作日志仍是“可管理的日志表”，不是最终审计方案
- 当前 `operation_log` 已有前后端模块，管理员可查看和维护
- 但真实项目里更合理的做法是：
- 用 AOP 或拦截器自动记录日志
- 自动采集方法、地址、IP、操作人、耗时、结果
- 操作日志默认只读
- 删除或清理改成归档式管理而不是普通 CRUD

## 五、后端结构化优化

### 1. 返回层仍可继续 VO 化
- 当前为了推进效率，很多接口仍直接返回实体或接近实体的结构
- 后续可逐步拆成：
- `ListVO`
- `PageVO`
- `DetailVO`
- 减少字段暴露与前后端耦合

### 2. 错误码与异常体系仍可继续细化
- 当前 `Result<T>` 已经够用
- 后续仍可继续补：
- 统一业务错误码
- 更明确的 message 规范
- 典型业务异常分类
- 更适合前端消费的错误结构

### 3. 数据库与约束仍可继续增强
- 当前主要依赖代码保证业务一致性
- 后续可逐表复盘：
- 是否增加联合唯一索引
- 是否引入更清晰的状态字典
- 是否保留现有“少外键、多代码控制”的策略
- 是否补归档字段或操作历史字段

## 六、文件链路优化

### 1. 当前文件存储仍是本地磁盘
- 这对本地开发和第一版部署是可接受的
- 后续若继续走生产化，可升级为：
- MinIO
- OSS
- 更明确的存储抽象层
- 文件迁移策略

### 2. 文件链路仍可继续补审计能力
- 当前已完成上传、预览、下载、删除联动
- 后续可继续补：
- 下载日志
- 预览日志
- 文件内容类型更严格校验
- 更清晰的目录规划与命名策略

## 七、当前优先级建议

### 第一优先级
- 前端性能优化
- 详情 loading、skeleton、empty 统一
- 部署准备文档
- 环境变量与打包上线流程

### 第二优先级
- 首页和少量页面视觉精修
- 用户管理字段扩展与批量导入
- 建组申请方案 C 继续产品化
- 操作日志自动采集

### 第三优先级
- VO 分层
- 错误码体系
- 文件存储升级
- 更细的安全与审计策略

## 当前结论
- 当前这些优化项都值得做，但绝大多数已经不再阻塞主业务使用
- 回归测试已经证明主业务链路可用，下一阶段最应该投入的是“性能观感”和“详情交互一致性”
- 如果把性能和加载体验这一轮做好，接下来再进入部署准备会更稳，也更适合作为作品展示与上线版本
