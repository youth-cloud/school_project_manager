# 可优化项记录

## 最新状态
- 当前主业务链已跑通，最近一轮全局 Code Review 已通过
- 核心业务模块已完成第一轮真实角色校验落地
- 以下项目仍属于“非阻塞但值得继续优化”的内容

## 使用说明
- 本文件用于记录“当前先不做，但后续值得优化”的内容
- 原则：先保证主线业务全部跑通，再回头逐项优化
- 每完成一个大模块后，可继续补充新的优化点
- 后续真正开始优化时，可按优先级逐项处理并打勾

---

## 一、整体架构层

### 1. 角色校验已完成第一轮落地，但仍需继续补强
- 当前核心业务模块已接入教师/学生/管理员角色校验
- 但仍需继续补齐剩余模块、登录态联动、接口级权限收口
- 后续需要结合 Spring Security / JWT 做统一权限控制

### 2. Spring Security 已完成第一轮收口，但仍需继续细化
- 当前已支持：登录接口放行、JWT 过滤器、未登录 JSON 响应、其他接口默认要求认证
- 后续仍需补：
- 更细的角色级访问控制
- 业务接口的权限边界细化
- 更完整的异常处理与前端联动

### 3. JWT 登录鉴权已完成第一轮闭环，但仍未完全结束
- 当前已完成：
- 登录接口
- token 生成
- token 解析
- 登录态获取当前用户信息
- `/api/auth/me`
- 后续仍需补：
- PasswordEncoder 密码加密
- token 刷新 / 退出登录策略
- 当前登录用户进一步接入业务接口

### 4. 操作日志、审计能力未接入
- `operation_log` 表已补齐
- 但目前还没有真正记录谁在什么时候做了什么操作
- 后续可结合 AOP 或拦截器统一记录

### 5. 数据库结构标准文件仍需持续维护
- 当前已经开始补齐缺失表
- 后续每次新增表或改字段，都要同步更新标准 SQL 文件
- 避免数据库结构和代码再次脱节

## 二、接口设计层

### 1. 目前更新接口统一使用“请求体带 id”
- 现在这种风格没问题，也便于教学和统一
- 但后续如果想更 RESTful，可考虑改为：
- `PUT /api/xxx/{id}`
- 当前不急，先保持全项目风格统一

### 2. 列表接口和分页接口并存，可后续考虑规范化
- 当前很多模块同时保留：
- 列表接口：`GET /api/xxx`
- 分页接口：`GET /api/xxx/page`
- 后续如果数据量变大，可考虑更多场景直接走分页接口

### 3. 统一返回结构仍可进一步细化
- 目前 `Result<T>` 已经够用
- 后续可增加：
- 更细的错误码枚举
- 更统一的业务码定义
- 更规范的 message 文案

### 4. DTO 可继续细分
- 目前已经做到了 CreateDTO / UpdateDTO / PageQueryDTO
- 后续还可以增加：
- DetailVO
- PageVO
- ListVO
- 避免后面直接把实体对象原样返回给前端

## 三、ProjectGroupMember 模块可优化项

### 1. 当前使用物理删除
- 这是因为表结构本身没有 `deleted` 字段
- 现阶段可接受
- 后续如需保留成员历史，可升级为：
- 状态退出
- 逻辑删除
- 或新增退出时间等字段

### 2. 未校验“提交成员是否真的属于该项目组”这一类更深层业务
- 当前已做：
- 项目组存在
- 用户存在
- 重复入组
- 组长唯一
- 后续可继续补：
- 成员人数上限控制
- 组长与 `project_group.leader_id` 的一致性控制

### 3. 组长信息存在双处表达风险
- 现在：
- `project_group.leader_id`
- `project_group_member.is_leader`
- 两边都能表达“谁是组长”
- 后续需要明确单一真相源，或者建立一致性校验

## 四、StageTask 模块可优化项

### 1. 截止时间合法性校验还不够
- 当前主要做了批次、教师、阶段序号校验
- 后续可增加：
- `deadline` 不早于批次开始时间
- `deadline` 不晚于批次结束时间

### 2. 阶段序号唯一性目前由代码校验
- 现在已经可用
- 后续可考虑数据库唯一索引和代码校验双保险
- 如果数据库已加唯一索引，也要补充异常友好提示

### 3. 阶段任务状态语义仍可细化
- 当前 `status` 是整数型基础状态
- 后续可考虑明确语义，如：
- 0 停用
- 1 进行中
- 2 已截止
- 3 已归档

### 4. needReport 等字段后续可抽象
- 当前用多个 `needXxx` 字段很直观
- 后续如果提交要求越来越多，可考虑做配置化或 JSON 扩展
- 当前阶段不急，保持直白更适合学习

## 五、StageSubmission 模块可优化项

### 1. 提交人是否属于该项目组尚未校验
- 当前已校验：
- 任务存在
- 批次存在
- 项目组存在
- 提交人存在
- 任务/项目组与批次一致
- 后续应补：
- `submitterId` 是否真的是该 `groupId` 下成员

### 2. 版本号目前由前端传入
- 当前这样写方便跑通流程
- 更合理的后续做法是：
- 后端自动根据同任务/同项目组的历史记录生成下一个版本号
- 避免前端随便传错版本号

### 3. 未校验“同任务同项目组版本号唯一”
- 当前还没做这条约束
- 后续可在代码层或数据库层增加唯一性约束

### 4. 提交时间当前只在新增时由后端赋值
- 这已经比前端传时间更合理
- 后续可区分：
- 保存草稿时间
- 正式提交时间
- 最后更新时间

### 5. 提交状态语义可继续细化
- 当前 `status` 只是基础整数状态
- 后续可规范成：
- 草稿
- 已提交
- 已审核
- 需重新提交
- 已通过
