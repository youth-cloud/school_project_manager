# 可优化项记录

## 当前说明
- 本文件只记录“当前先不做，但后续值得继续优化”的内容
- 现在项目已完成主业务第一版、主要业务纠偏、完整回归测试，以及最后一轮业务封板扫尾
- 优化项不再按“先补功能模块”排序，而按“更接近可交付作品”的真实优先级重新整理
- 当前最值得投入的方向已经从“继续补 CRUD”转为“部署演练 + 文档对齐 + 前端体验统一 + 最终美化”

## 一、当前最高优先级

### 1. 完整部署演练
- 当前部署资源已经具备第一版基础：
- `Dockerfile`
- `docker-compose.yml`
- `nginx.conf`
- `.env.example`
- `部署说明.md`
- 但还没有真正完成一次“从空环境到系统可访问”的完整演练
- 后续建议逐项验证：
- MySQL 初始化是否可直接跑通
- Redis 是否按当前认证链路正确接入
- 文件上传目录是否在容器和宿主机之间正确映射
- 前后端环境变量是否和本地开发环境一致
- Nginx 反向代理、静态资源和上传文件访问是否正常
- 云服务器上从构建到启动的步骤是否无需人工补洞

### 2. 文档对齐与交付文档整理
- 现在业务状态已经和早期文档有明显差异，后续需要把交付口径完全整理清楚
- 建议继续补齐或更新：
- 部署步骤里的服务器准备、端口、安全组、域名配置
- 数据库初始化和演示数据导入方式
- 文件目录说明与备份建议
- 默认账号或演示账号说明
- 回归测试结论、已知限制、上线前检查清单
- 最终对外演示时建议准备一份更精炼的“项目使用说明”

### 3. 前端性能优化
- 当前数据量很小的情况下，部分页面首屏仍会出现 `1~2s` 的 `no data` 或空白等待
- 后续仍应优先排查：
- 是否存在页面 `onMounted` 串行请求过多
- 是否存在详情弹窗“先开弹窗再等多次接口返回”的模式
- 是否存在无必要的全量 `options` 重复请求
- 是否存在查询条件和详情数据分开多次请求但没有并行化
- 建议优先优化的页面：
- `project_topic`
- `topic_application`
- `project_group_application`
- `project_group`
- `project_group_member`
- `stage_submission`
- `review_record`
- `defense_record`

### 4. 详情 loading / skeleton / empty 体验统一
- 当前只有部分页面补了加载态和骨架屏，整体体验仍不完全统一
- 后续建议统一出一套规则：
- 列表首屏使用统一 loading 状态
- 详情弹窗打开后先显示 skeleton，而不是空白框
- 空列表、空附件、空详情采用统一 empty 组件文案
- “提交中 / 删除中 / 加载中”按钮状态风格统一
- 这一项做完后，前端观感会明显更像成品系统

## 二、前端产品化与视觉优化

### 1. 最终界面美化
- 用户已经明确：当项目没什么明显优化空间后，进入界面美化阶段
- 当前业务封板已接近完成，界面美化已经从“遥远计划”变成下一大阶段候选项
- 后续可重点优化：
- 首页卡片布局和信息密度
- 列表页标题区和操作区的一致性
- 详情弹窗卡片层级
- 蓝白主视觉细节
- 图标、间距、阴影、按钮层次

### 2. 列表页与详情页继续通用化
- 当前很多页面已经形成统一模板，但实现仍分散
- 后续可继续抽：
- 通用分页状态
- 通用详情 loading / skeleton
- 通用状态标签映射
- 通用时间格式化
- 通用确认删除逻辑
- 通用详情卡片布局

### 3. 页面中仍可能残留零碎教学文案或过渡性表达
- 当前已做过一轮大清理
- 后续仍建议在最终交付前再统一扫一遍：
- 页头副标题
- 暂时性说明
- 教学提示文案
- 开发期标签
- 接口说明式文字

### 4. 选择器体验仍可继续升级
- 当前很多纯 ID 已经改成名称与选择器
- 后续可继续补：
- 远程搜索
- 大数据量分页搜索
- 选项缓存
- 联动选择器
- 更明确的空态与加载态

## 三、业务规则层优化

### 1. 建组申请与正式项目组链路继续产品化
- 当前主流程已跑通：学生申请建组、教师审核、系统自动生成正式项目组
- 当前也已支持：
- 申请人默认成为组长
- 其他成员通过输入项目组 ID 直接加入
- 后续可继续补：
- 建组申请与正式项目组跳转联动
- 更清晰的审核后反馈
- 申请历史状态追踪
- 组 ID 分享入口的进一步精修

### 2. StageSubmission 仍可继续增强业务一致性
- 当前已完成提交链路校验、附件整合、删除保护和父级关系固定化
- 后续仍可补：
- `submitterId` 是否真属于该项目组成员的更细校验
- `versionNo` 改为后端自动生成
- 同任务同项目组版本唯一性
- 提交状态进一步产品化
- 教师查看同一组多版本提交的差异化体验

### 3. WeeklyReport 仍可继续流程化
- 当前周报第一版已可用，且已经完成同组可见和父级关系固定化
- 后续仍可补：
- 同一学生同一周次唯一性
- 教师点评与学生提交拆成独立动作
- 周报状态流转更清晰
- 周报列表的时间维度筛选体验

### 4. DefenseSchedule / DefenseRecord / ScoreRecord 仍可继续规则化
- `defense_schedule` 后续可补：
- 答辩日期与批次时间范围约束
- 同时间地点冲突校验
- 同批次顺序唯一性
- `defense_record` 后续可补：
- 评审教师资格约束
- 重复评分策略
- 总分后端自动计算
- 分项上限与权重规则
- `score_record` 后续可补：
- `finalScore` 与 `gradeLevel` 后端自动生成
- 更明确的成绩发布状态
- 成绩查看与公布流程区分

## 四、认证、安全与审计优化

### 1. 认证生命周期继续增强
- 当前已完成：登录、`/me`、改密、管理员重置密码、refresh、logout、旧 token 失效控制
- 后续可继续补：
- 双 token 模型
- 多设备登录策略
- 强制下线指定用户
- 登录失败次数限制
- 风险登录提示
- 更细的 token 生命周期配置化

### 2. 用户选项接口按场景裁剪
- 当前已经修复过“不同身份显示不一致”的问题，也抽了公共名称映射能力
- 后续仍可继续优化为：
- 不同页面只返回当前页面真正需要的候选人
- 复杂场景改成分页搜索型选择器
- 避免长期依赖通用全量 options 接口

### 3. 操作日志升级为更真实的审计方案
- 当前 `operation_log` 已有前后端模块，管理员可查看和维护
- 但真实项目里更合理的做法是：
- 用 AOP 或拦截器自动记录日志
- 自动采集方法、地址、IP、操作人、耗时、结果
- 操作日志默认只读
- 删除或清理改成归档式管理而不是普通 CRUD

## 五、后端结构化优化

### 1. 返回层继续 VO 化
- 当前为了推进效率，很多接口仍直接返回实体或接近实体的结构
- 后续可逐步拆成：
- `ListVO`
- `PageVO`
- `DetailVO`
- 减少字段暴露与前后端耦合

### 2. 错误码与异常体系继续细化
- 当前 `Result<T>` 已经够用
- 后续仍可继续补：
- 统一业务错误码
- 更明确的 `message` 规范
- 典型业务异常分类
- 更适合前端消费的错误结构

### 3. 数据库与约束继续增强
- 当前主要依赖代码保证业务一致性
- 经过这轮封板后，已经比较容易看出哪些规则值得下沉到数据库
- 后续可逐表复盘：
- 是否增加联合唯一索引
- 是否引入更清晰的状态字典
- 是否保留现有“少外键、多代码控制”的策略
- 是否补归档字段或操作历史字段

## 六、文件链路优化

### 1. 当前文件存储仍是本地磁盘
- 这对本地开发和第一版部署是可接受的
- 后续若继续走生产化，可升级为：
- MinIO
- OSS
- 更明确的存储抽象层
- 文件迁移策略

### 2. 文件链路继续补审计能力
- 当前已完成上传、预览、下载、删除联动
- 后续可继续补：
- 下载日志
- 预览日志
- 文件内容类型更严格校验
- 更清晰的目录规划与命名策略

## 七、当前优先级建议

### 第一优先级
- 完整部署演练
- 部署说明与交付文档整理
- 前端性能优化
- 详情 loading、skeleton、empty 统一

### 第二优先级
- 最终界面美化
- 首页和少量页面视觉精修
- 建组申请链路继续产品化
- 用户管理字段扩展与批量导入

### 第三优先级
- VO 分层
- 错误码体系
- 操作日志自动采集
- 文件存储升级
- 更细的安全与审计策略

## 当前结论
- 当前这些优化项都值得做，但绝大多数已经不再阻塞主业务使用
- 主业务、权限边界、删除保护和关系固定化已经完成关键收口，当前项目已经具备进入部署演练和最终整理的条件
- 下一阶段最应该投入的是“部署可落地性 + 前端观感统一 + 最终作品级呈现”，而不是继续无止境补基础 CRUD
